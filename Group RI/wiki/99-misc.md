## วัตถุประสงค์ของระบบ (Purpose)
ระบบ Group RI ถูกออกแบบเพื่อรองรับกระบวนการ ประมวลผลข้อมูลประกันภัยต่อ (Reinsurance) ทั้งในส่วนของ
- Estimate Process (ข้อมูลประมาณการ) และ
- Actual Process (ข้อมูลจริง)
โดยมีจุดมุ่งหมายหลักเพื่อ :
- ประมวลผลเบี้ย สินไหม และ คอมมิชชั่น ตาม Treaty ของ Reinsurer แต่ละราย
- สร้างผลลัพธ์ในรูปแบบ Report และ ไฟล์ EDW / SOA สำหรับบัญชีและรายงานประกันภัยต่อ
- เพิ่มประสิทธิภาพในการตรวจสอบ ลดขั้นตอน manual และ ควบคุมคุณภาพข้อมูล
⚙️ ภาพรวมกระบวนการระบบ (System Overview)
ระบบ Group RI แบ่งการทำงานออกเป็น 2 ส่วนหลักคือ
- Estimate Process – การประมวลผลข้อมูลประมาณการ (รายเดือน)
- Actual Process – การประมวลผลข้อมูลจริง (รายไตรมาส)

## ⚙️ ภาพรวมกระบวนการของระบบ (Overall Processing Flow)
| ลำดับ | ขั้นตอนหลัก | รายละเอียดโดยย่อ |
| 1 | การดึงข้อมูล (Data Pull) | ดึงข้อมูลกรมธรรม์ สินไหม และข้อมูลเสริมจากระบบต้นทาง (GI System / Claim SQL) แยกตาม Treaty ที่เลือก |
| 2 | การประมวลผล (Processing) | ระบบคำนวณ Estimate หรือ Actual ตามสูตรของแต่ละ Treaty และตรวจสอบความถูกต้องของข้อมูล |
| 3 | การตรวจสอบและยืนยัน (User Review & Confirmation) | User RI ตรวจสอบผล Download Excel ตรวจ และยืนยัน |
| 4 | การแปลงและส่งข้อมูล (Convert to Standard Template & Export) | ระบบสร้าง Standard Template และ ส่งเข้า EDW และ I-Master |
| 5 | การบันทึกบัญชีและรายงาน (Accounting & Reporting) | ข้อมูลจาก EDW ใช้สำหรับบันทึกบัญชี และจัดทำ R-Report (R18 – R20) |

## 1. กระบวนการ Estimate (Estimated Report Flow – TO BE)
กระบวนการนี้ใช้สำหรับการประมวลผลข้อมูลประมาณการรายเดือน โดย User RI เป็นผู้ดำเนินการหลัก

### ขั้นตอนหลัก

#### 1. การดึงข้อมูล
- ดึงข้อมูลจากระบบต้นทาง ได้แก่ ระบบประกันกลุ่ม (Group Policy) และ Claim SQL
- เลือกประมวลผลได้ทั้ง ราย Treaty หรือ ทุก Treaty
- ข้อมูลที่ใช้ เช่น List of Policy, ข้อมูลกรมธรรม์ต้นสัญญา, Claim (เฉพาะที่ Approved ใน Period ที่เลือก), เบี้ย, Layer, รายสมาชิก เป็นต้น

#### 2. การประมวลผลข้อมูล
- ระบบจะประมวลผลตามสูตรเฉพาะของแต่ละ Treaty เช่น Estimate_Daiichi_Calculation, Estimate_Gibraltar_Calculation, Est_Thaire_Calculation
- หลังรันเสร็จ ระบบแจ้งผลผ่าน Auto-Mail ให้ User ตรวจสอบ
- Output: ไฟล์สำหรับ EDW, SOA, I-Master, Error log

#### 3. การตรวจสอบและยืนยันข้อมูล
- ผู้ใช้ตรวจสอบผลผ่านหน้าจอ และสามารถ Download Excel เพื่อตรวจสอบ
- หากพบข้อมูลผิด แจ้ง HelpDesk และประมวลผลซ้ำได้
- เมื่อถูกต้อง ยืนยันเพื่อเข้าสู่ขั้นตอน Convert Standard Template

#### 4. การแปลงเป็น Standard Template และส่งข้อมูลปลายทาง
- Convert เป็น Standard Template เพื่อให้เป็นรูปแบบกลาง
- ส่งข้อมูลเข้าสู่ EDW และ I-Master (เพื่ออัปเดต RI Information ใน Monthly Policy Snapshot)

#### 5. การบันทึกบัญชีและรายงาน
- ข้อมูลใน EDW ใช้โดยฝ่ายบัญชีในการ Generate Ref No. และบันทึกบัญชี
- ฝ่ายคณิตศาสตร์จัดทำรายงาน R18 / R19 / R20
- ข้อมูลสุดท้ายถูกเก็บใน I-Master

| Treaty | ข้อมูลที่ใช้หลัก | เงื่อนไขการคำนวณ | หมายเหตุ |
| Dai-ichi Group EB (PS-01) | List of Policy, ข้อมูลต้นสัญญา, Claim, Estimate Premium | ใช้สูตรคำนวณเบี้ยตาม Payment Mode (Annual×1, Monthly×12, Quarterly×4, Semi-Annual×2) | Round(Premium/2,2) |
| Gibraltar Group EB (PS-02) | List of Policy, ข้อมูลต้นสัญญา, Claim, Layer | คำนวณแยก Layer (≤5M / 5–20M / >20M) | ใช้สำหรับ RI Premium Life/Accident |
| Thaire Group PA (PS-03) | List of Policy, ข้อมูลต้นสัญญา, Claim, รายสมาชิก | คำนวณรายสมาชิก โดยพิจารณา Retention, Quota Share, Surplus | แบ่งช่วง SA: ≤0.4M, ≤2M, ≤10M |
เงื่อนไขเลือกกรมธรรม์ (ทุก Treaty):
- กรมธรรม์ที่ Effective Date = Period และ Active > 0
- กรมธรรม์ที่ Effective Date < Period แต่ยัง Active และยังไม่เคยประมวลผล
- กรมธรรม์ที่ Effective Date > Period → ไม่ถูกประมวลผล

## 2. กระบวนการ Actual (Gibraltar EB & Thaire PA)
เพื่อประมวลผลข้อมูลจริงรายไตรมาส (Quarterly Actual) ของสัญญา Reinsurance กลุ่ม Gibraltar EB และ Thaire PA โดยระบบ Group RI จะคำนวณ เบี้ย สินไหม คอมมิชชั่น และ กำไรจากการประกันภัยต่อ ตามเงื่อนไขเฉพาะของแต่ละ Treaty และสร้าง Report สำหรับ Reinsurer รวมถึงไฟล์นำเข้า EDW และ SOA

## ⚙️ ภาพรวมกระบวนการทำงาน
- เลือก Treaty และ Quarter Period (ย้อนหลังได้ 2 ปี)
- ดึงข้อมูลจากต้นทาง (GI System / Claim SQL) เช่น List of Policy, List of Inforce, Premium, Claim, Alteration, Layer Data ฯลฯ
- คำนวณผลตาม RI Condition ของแต่ละ Treaty (Accident / Life / Layer / Member ระดับรายบุคคล)
- ตรวจสอบ ผล Download Excel และ ยืนยัน ก่อนสร้าง Standard Template
- ส่งออก Output File สำหรับ Reinsurer , EDW และ SOA
- คำนวณ Profit Commission (เฉพาะ Quarter 4) และจัดทำรายงานสรุป

## 📊 Business Rules หลักของแต่ละ Treaty
| หัวข้อ | Gibraltar Group EB | Thaire Group PA |
| ความคุ้มครอง | Life / Accident / Dismemberment / TPD | Accident Death (AD&D) เท่านั้น |
| Retention | ตาม Layer ทุน (1–3 ชั้น) | 400,000 บาท |
| Minimum Sum Assured | SA ≥ 1 ล้านบาท (แบ่ง Layer ตาม 5 M และ 20 M) | SA ≥ 500,000 บาท |
| โครงสร้าง Layer / Share | L1 ≤ 5 M → 15% share L2 5–20 M → 100% L3 > 20 M → 0% | 400k–2 M → Quota 50% 2–10 M → Surplus 100% |
| RI Rate / Commission | ตาม Age และ Payment Mode | RI Rate ตาม Age Commission 25% x Premium |
| ส่วนลด | ไม่มี (คำนวณตรงตาม Premium Rate) | ส่วนลด M&A 0–20% และ Group Volume 10–35% |
| Movement ที่คำนวณ | New Business / Renewal / Revivals / Refund / Claim | NB / Renewal / Revivals / Refund / Claim (ใช้ Alteration Sheet ราย Member) |
| Claim & Expense | คำนวณ Reinsurer’s Share ตาม SA และ Layer | ใช้ Reinsurer’s Share ตาม Claim Sheet ของ สมาชิก |
| Profit Commission | Quarter 4 เท่านั้น (คำนวณ 50% ของ กำไรสุทธิ ถ้ามีกำไร) | Quarter 4 เท่านั้น และเฉพาะกรณีมี Member > 200 ตลอดปี |

## 🧩 Input Data ที่ใช้ร่วมกัน
- List of Policy
- List of Inforce
- Claim (เฉพาะ Accident / TPD / Dismemberment)
- Premium / Movement (Report R61 หรือ Alteration_Calculation.xlsx)
- Layer / OverAge Data (สำหรับ Gibraltar)
- Alteration / Member Level Data (สำหรับ Thaire)

## 🧭 สรุปภาพรวม Actual
การประมวลผล Actual Group RI ของ Gibraltar EB และ Thaire PA มีเป้าหมายเดียวกันคือจัดทำข้อมูลจริงสำหรับ Reinsurer และ ระบบบัญชีภายใน โดยต่างกันที่รูปแบบความคุ้มครองและโครงสร้าง Layer / Retention ที่ใช้คำนวณ แต่ทั้งสอง Treaty ใช้หลักการประมวลผล Quarterly และมีการคำนวณ Profit Commission ใน Quarter 4 เท่านั้น
