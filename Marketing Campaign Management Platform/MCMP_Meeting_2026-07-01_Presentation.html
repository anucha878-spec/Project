2ที่เก็บข้อมูล & การจำแนก Customer / Prospect

🗄️ CX = ที่เก็บข้อมูลลูกค้า (CDP กลาง)

  • ข้อมูล Customer ให้ลงที่ CX เป็นหลัก (ทิศทางจากพี่โทน)
  • แยกชุดข้อมูลใน CX โดยมีชุดหนึ่งเป็นฝั่ง Campaign / Activity
  • ลูกค้าทำธุรกรรม (recurring) → บันทึกเข้า CX ไม่ใช่ upload ครั้งเดียวจบ

👥 จำแนก 2 ทาง: Customer vs Prospect

  • Customer → เก็บที่ CX ต้อง confirm: CX = TCM?
  • Prospect ยังไม่ล็อก — 2 แนวทาง:
  •   (ก) ขยาย CX ให้รองรับ Prospect → ตามรอย convert เป็นลูกค้าได้
  •   (ข) เก็บ DB แยก (TCM / marketing DB) ต้อง confirm
QA note: ต้องมี test data + assertions แยก 2 กลุ่มชัดเจน และตรวจว่าแต่ละกลุ่มถูกจัดเข้าที่เก็บถูกต้อง

3Validation ตอน Ingest ข้อมูล

[ ข้อมูลเข้า ] [ Validate Gate ] [ ปลายทาง ] Upload / Feed ──► ✔ Identify "เป็นใคร" ┌─► ✅ ผ่าน → CX (ลูกค้า/Lead) ✔ เลขบัตร ปชช. 13 หลัก │ └─ ถ้าตรงลูกค้าเดิม → link/merge ✔ ตรวจความถูกต้อง (valid) │ └─► ❌ ไม่ผ่าน → reject / drop ออก

กติกาที่ระบุในประชุม

  • ยืนยันตัวตน ไม่รับข้อมูลขยะ / ชื่อมั่ว
  • เลขบัตรประชาชน 13 หลัก + ตรวจ valid ต้อง confirm rule เต็ม
  • Record ไม่ผ่าน → คัดออก ไม่เข้า CX
  • ผ่านแล้วตรงลูกค้าเดิม → link/merge

QA ต้องเตรียม

  • Positive / Negative validation test
  • Boundary เลขบัตร (12/13/14 หลัก, checksum)
  • Duplicate / merge scenario
  • Reject-flow verification (ข้อมูลหลุดเข้าไหม)

4ผู้ใช้คือการตลาด → ไม่ให้เข้า CX ตรง

[ ผู้ใช้: ทีมการตลาด ] [ ระบบเบื้องหลัง ] ┌───────────────────────┐ deliver / ┌──────────────┐ │ UI / เมนู │ ───── เชื่อมโยง ─────► │ CX │ │ บน Marketing Platform │ ข้อมูล │ (CDP กลาง) │ └───────────────────────┘ └──────────────┘ ผู้ใช้ทำงานที่นี่ ไม่ให้การตลาดเข้า CX โดยตรง
  • คนใช้งานจริง = ฝ่ายการตลาด → เข้าไปทำงานที่ CX โดยตรง "ไม่เหมาะ"
  • ดีไซน์: สร้าง UI/เมนูบน marketing platform ส่วนข้อมูล deliver ไป CX เบื้องหลัง
  • ค่าตั้งต้น: "ออกแบบให้ไปที่ CX" แล้วค่อยพิจารณาเมนูรายตัว
QA note: ต้องทดสอบ integration/data-integrity — UI บน marketing tool → ข้อมูลลง CX ถูกต้องครบถ้วน
