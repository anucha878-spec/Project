Overview

ใช้สำหรับการเช็คข้อมูลในตาราง tx_tmp_commission_rate ว่าเคยมีข้อมูลอยู่แล้วหรือไม่ กรณีมีการเพิ่มข้อมูล commission ของแต่ละประเภท

## Process

1. [เช็ค Effective Date From และ Effective Date To ต้องอยู่ในช่วงของ Treaty](/pages/viewpage.action?pageId=1103266503) หากอยู่ในช่วง ให้ข้ามไปทำข้อ 2. หากไม่อยู่ในช่วงให้ return ดังนี้NameTypeExampleresponseCodeString102responseMessageStringINVALID_EFFECTIVE_DATE
2. นำ tmp_treaty_id ไปค้นหาที่ตาราง tx_tmp_comission_rate ว่าพบข้อมูลหรือไม่ <![CDATA[SELECT * FROM tx_tmp_comission_rate WHERE tmp_treaty_id = :tmpTreatyId]]>  ถ้าไม่พบข้อมูล สามารถ insert ลงตาราง tx_tmp_comission_rate และตารางลูกๆตาม type ได้เลย เช่น type Policy Year ให้บันทึกลงตาราง tx_tmp_comm_by_yearถ้าพบข้อมูล ให้ตรวจสอบดังต่อไปนี้ให้ตรวจสอบว่า ฟิล์ด ms_comm_type ของ input เข้ามาใหม่ ตรงกันกับข้อมูลเดิมที่อยู่ในตาราง tx_tmp_comission_rate หรือไม่กรณีตรงกัน (ไม่เปลี่ยนข้อมูลในตาราง tx_tmp_comission_rate) ให้ทำการเช็คเงื่อนไขต่อไปนี้ เช็คว่า effective date ที่ input เข้ามาใหม่นั้นทับซ้อนกันกับข้อมูลเดิมหรือไม่ โดยเช็คกับตารางลูกๆของแต่ละ type กรณีที่เช็คแล้ว effective date ยังไม่เคยมี ให้ทำการ insert ข้อมูลลงตารางลูกๆของแต่ละ type ได้เลยกรณีที่เช็คแล้ว effective date เคยมีข้อมูลอยู่แล้ว ให้ return ค่าดังนี้NameTypeExampleresponseCodeString102responseMessageStringINVALID_EFFECTIVE_DATEกรณีที่ไม่ตรงกัน (มีการปรับแก้ไขข้อมูลในตาราง tx_tmp_comission_rate) ให้ทำการเช็คเงื่อนไขต่อไปนี้ให้เช็คว่าตารางลูกๆของเดิม เคยมีข้อมูลเก่าอยู่แล้วหรือไม่ เช่น เดิมมีการบันทึกข้อมูล type Policy Year เคยมีข้อมูลอยู่ในตาราง tx_tmp_comm_by_year และเปลี่ยน type เป็น Policy Term ต้องทำการเช็คก่อนว่า มีข้อมูลอยู่ในตาราง tx_tmp_comm_by_year ค้างอยู่หรือไม่ หากมีค้างอยู่ จะไม่สามารถบันทึกข้อมูลได้ ให้ทำการ return ค่าดังนี้NameTypeExampleresponseCodeString100responseMessageStringINVALIDกรณีเช็คแล้วว่า ข้อมูลของตารางลูกๆเดิม ไม่มีข้อมูลอยู่แล้ว จะสามารถ อัพเดทตาราง tx_tmp_commission_rate เพื่อเปลี่ยน type และบันทึกข้อมูลใหม่ลงตารางลูกๆได้
