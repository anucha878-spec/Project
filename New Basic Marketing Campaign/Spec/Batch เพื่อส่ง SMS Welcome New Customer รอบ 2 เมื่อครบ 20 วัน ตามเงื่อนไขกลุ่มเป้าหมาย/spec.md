## ภาพรวม (Business Context)

Batch process จัดส่ง SMS "Welcome New Customer (Ocean Club)" เป็น **แคมเปญรอบที่สอง (follow-up)** ต่อจาก SMS "Welcome New Customer (LINE)" ของ CSMS-001 — ระบบจะค้นหาลูกค้าใหม่ที่เคยได้รับ SMS รอบแรกครบกำหนด 20 วัน (ปรับค่าได้) กรมธรรม์ยังมีผลบังคับ (Inforce) และ **ยังไม่ได้ลงทะเบียน** ทั้ง LINE Ocean Connect และ Ocean Club App แล้วส่ง SMS เชิญชวนให้ดาวน์โหลดและลงทะเบียน Ocean Club App พร้อมสิทธิพิเศษ (เหรียญรางวัล) เพื่อสร้างความประทับใจ ยืนยันความคุ้มครอง และเพิ่มยอดผู้ใช้งานแอปพลิเคชัน ครอบคลุมกรมธรรม์ประเภทสามัญ (ORD), อุตสาหกรรม ปช. (IND), อุตสาหกรรม ขพ. (GOV) และอุบัติเหตุส่วนบุคคล (PA)

**ผู้เกี่ยวข้อง (Roles)**

| Role                                        | สิทธิ์                                  | ขอบเขต                                                                |
| ------------------------------------------- | --------------------------------------- | --------------------------------------------------------------------- |
| `SYSTEM_BATCH` — ระบบ Batch (Scheduled Job) | ประมวลผลอัตโนมัติ                       | ลูกค้าทุกรายที่เข้าเงื่อนไข                                           |
| `IT_ADMIN` — ผู้ดูแลระบบ                    | Manual Trigger / Re-run เฉพาะ Batch Fix | เลือกช่วงวันที่ที่ต้องการซ่อมข้อมูล ผ่าน Admin Dashboard กลางของ CSMS |

**ขอบเขต (Scope)**

ใน scope: ดึงกลุ่มเป้าหมายจากประวัติการส่ง SMS รอบแรก (CSMS-001) ที่ครบกำหนดจำนวนวันที่ตั้งค่าไว้ → ตรวจสถานะกรมธรรม์ Inforce และเบอร์โทรศัพท์ → คัดกรอง Do Not Contact List → ตรวจสถานะการลงทะเบียน LINE Ocean Connect / Ocean Club App → ตัดรายการซ้ำ → สร้างไฟล์ CSV → ส่ง SMS ผ่านช่องทางกลาง (ESB → SMS Gateway) → บันทึก Log 2 ระดับพร้อมการเชื่อมโยงกลับรายการรอบแรก → แจ้งเตือนเมื่อล้มเหลว → รองรับ Manual Fix

นอก scope: การออกแบบ UI/Field ใหม่ของ Admin Dashboard (ใช้หน้า Manual Fix กลางของ CSMS ที่มีอยู่แล้ว เพียงเพิ่ม batch นี้เข้า list — แนวทางเดียวกับ CSMS-001), การออกแบบรายงานใหม่ (ใช้รายงานสรุปกลางของ CSMS ที่ใช้ร่วมกันทุกแคมเปญ เพียงเพิ่มแคมเปญนี้เข้าไป), รายละเอียด logic การคำนวณ credit_amount (ใช้ logic เดิมของ CSMS), การเปลี่ยนแปลงใดๆ ต่อ batch CSMS-001 (รอบแรก)

### Session 2026-07-07

- Q: เอกสารต้นทาง Pre-event ระบุครอบคลุมเฉพาะ ORD, IND, PA แต่เงื่อนไข Inforce/Do Not Contact มีประเภท GOV ด้วย — ขอบเขตแคมเปญนี้รวม GOV หรือไม่? → A: **รวม GOV** — ครอบคลุม 4 ประเภท: ORD, IND, GOV, PA เงื่อนไขคัดกรองใช้ตามที่เอกสารต้นทางกำหนดครบทุกประเภท
- Q: วงเล็บ "(หรือ channel = 'LINE')" ในเกณฑ์ส่ง SMS ข้อ 1 ของเอกสารต้นทางหมายความว่าอย่างไร? → A: `code = 'E02'` คือสถานะ **"ไม่พบข้อมูล"** ใช้รหัสเดียวกันทั้งการตรวจ channel 'LINE' และ 'APP' — วงเล็บเป็นเพียงคำกำกับว่าผล E02 นั้นมาจากการเรียกตรวจฝั่ง LINE ไม่ใช่เงื่อนไขเพิ่มเติม เกณฑ์ส่งคงเป็น (LINE=E02 และ APP=E02) หรือ (LINE=E00 + isBlockLine=true และ APP=E02)
- Q: การ map รหัสประเภทกรมธรรม์จากเงื่อนไขคัดกรองเป็นค่าที่เก็บในตาราง log เป็นอย่างไร? → A: **Map ตรงตัว**: 'O'→ORD, 'I'→IND, 'G'→GOV, 'P'→PA (หมายเหตุ: เอกสารต้นทางแสดงรหัสคัดกรองเป็นเลข '0'/'1' สำหรับ ORD/IND — ยืนยันตัวอักขระจริงของรหัสในระบบต้นทาง ('O'/'I' หรือ '0'/'1') กับโครงสร้างข้อมูลจริงในขั้น plan โดยหลักการ map คือ หนึ่งรหัสต่อหนึ่งค่า log ตรงตัวครบ 4 ประเภท)
- Q: ลูกค้าคนเดียว-เบอร์เดียว มีหลายกรมธรรม์เข้าเงื่อนไขในรอบเดียวกัน ควรส่งกี่ข้อความ? → A: ยึดพฤติกรรมเดียวกับ CSMS-001 (spec 002): จัดกลุ่มตาม (ชื่อ, นามสกุล, เบอร์โทร) ส่งเพียง **1 ข้อความต่อลูกค้าต่อรอบ** โดยเลือกรายการตัวแทนแบบ deterministic (ตามแนว FR-009/A-007 ของ spec 002 — เลือกรายการเก่าสุด, tie-break ด้วย policy_no น้อยสุด) และลูกค้า 1 ราย ได้รับ SMS แคมเปญนี้ไม่เกิน 1 ข้อความตลอดแคมเปญ
- Q: เบอร์โทรที่ใช้ส่งจริง ใช้เบอร์จากประวัติรอบแรก (CSMS-001) หรือเบอร์ปัจจุบันจากข้อมูลกรมธรรม์? → A: **เบอร์ปัจจุบันจากข้อมูลกรมธรรม์** (`ili_policy_master`: mobile1 เป็นหลัก หากว่างใช้ mobile2) — เบอร์จากประวัติรอบแรกใช้เพื่อการอ้างอิง/ตามรอยเท่านั้น เพื่อลดโอกาสส่งไปเบอร์เก่าที่ลูกค้าเปลี่ยนแล้ว
- Q: รายงานสรุปผลประจำวัน (reconcile report) ของแคมเปญนี้จัดทำอย่างไร? → A: **ใช้รายงานกลางของระบบ CSMS ที่ใช้ร่วมกันทุกแคมเปญ** โดยเพิ่มแคมเปญนี้เข้าไปเป็นอีกหนึ่งรายการ — ไม่สร้างรายงานแยกใหม่
- Q: ปริมาณกลุ่มเป้าหมายคาดการณ์ต่อรอบประมวลผลเท่าใด? → A: **ประมาณ 1,000 รายต่อวัน**
- Q: การจัดการ error / timeout / การบันทึก log ใช้แนวทางใด? → A: **ใช้แนวทางเดียวกับ 002-batch-welcome-new-customer-line-sms ทั้งชุด**: E13/ผลนอกเงื่อนไข → skip รายรายการทันที ไม่ retry ไม่หยุดรอบ + บันทึก sent_status='F' (002: FR-008/FR-008a); บริการล่มทั้งระบบ → retry ≤3 ครั้งเว้นช่วง แล้วหยุดรอบ + email (002: FR-008b); บันทึกทุกรายการที่เข้าขั้นตอนส่ง/ถูก skip ด้วย sent_status 'S'/'F' โดยเพิ่มฟิลด์ sent_status ใน WELCOME_NEW_CUST_APP (002: FR-016a); ไม่พบชื่อลูกค้า → skip + 'F' ไม่ส่งข้อความไร้ชื่อ (002: FR-011a); email มาตรฐาน CSMS ระบุชื่อ batch และขั้นตอนที่ล้มเหลว เกณฑ์เวลาใช้ SLA มาตรฐาน monitoring CSMS (002: FR-018); รายการ 'F' ไม่ retry อัตโนมัติในรอบ daily ซ่อมผ่าน Manual Fix เท่านั้น (002: FR-020b)
- Q: กรณีไม่พบ cardNo จากหมายเลขกรมธรรม์ ควรคัดออกเงียบตามเอกสารต้นทาง หรือ skip + บันทึกรายรายการ? → A: **skip + บันทึก sent_status='F' รายรายการ** (แนวเดียวกับ 002 EC-005) — เพราะลูกค้าที่เคยรับ SMS รอบแรกควรมี cardNo เสมอ การไม่พบเป็นสัญญาณข้อมูลผิดปกติที่ต้องตรวจสอบย้อนหลัง/Manual Fix ได้
- Q: Manual Fix รันย้อนหลังใช้ค่าจำนวนวัน (N) ค่าไหนหากลุ่มเป้าหมายของรอบเก่า? → A: **ใช้ค่า N ปัจจุบันจาก config ณ เวลาสั่ง Manual Fix** — ไม่เก็บประวัติค่า config (YAGNI); หาก N เคยถูกเปลี่ยน IT Admin ปรับช่วงวันที่ที่เลือกชดเชยได้ และกลไกกันส่งซ้ำป้องกันการส่งเบิ้ลอยู่แล้ว

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ระบบส่ง SMS ชวนลงทะเบียน Ocean Club App อัตโนมัติรายวัน (Priority: P1)

ในฐานะระบบ (`SYSTEM_BATCH`) ต้องการค้นหาลูกค้าที่เคยได้รับ SMS "Welcome New Customer (LINE)" ครบกำหนด 20 วัน (ตามค่าที่ตั้งไว้) กรมธรรม์ยังมีผลบังคับ และยังไม่ได้ลงทะเบียนทั้ง LINE Ocean Connect และ Ocean Club App แล้วส่ง SMS เชิญชวนพร้อมชื่อลูกค้า มูลค่าเหรียญรางวัล และลิงก์ดาวน์โหลดแอป โดยอัตโนมัติทุกวันเวลา 10:00 น.

**Why this priority**: เป็นแก่นของ feature ทั้งหมด — หากไม่มีการคัดกรองและส่ง SMS อัตโนมัติ feature นี้ไม่มีคุณค่าทางธุรกิจ ส่วนอื่น (Manual Fix, แจ้งเตือน) เป็นส่วนเสริมของ flow หลักนี้

**Independent Test**: เตรียมข้อมูลประวัติการส่งรอบแรก (CSMS-001) ที่ `sms_sent_date` ครบกำหนด 20 วันพอดี ณ วันทดสอบ พร้อมกรมธรรม์สถานะ Inforce แล้วรัน batch หนึ่งรอบ — ตรวจว่า SMS ถูกส่งไปยังเบอร์ที่ถูกต้องด้วยข้อความที่แทนค่าตัวแปรครบถ้วน และมีการบันทึกประวัติพร้อมการเชื่อมโยงกลับรายการรอบแรก โดยไม่ต้องพึ่ง User Story อื่น

**Acceptance Scenarios**:

1. **Given** ลูกค้ามีประวัติรับ SMS รอบแรกครบ 20 วันพอดี ณ วันประมวลผล กรมธรรม์ ORD สถานะ Inforce มีเบอร์โทรศัพท์ ไม่ติด Do Not Contact และผลตรวจสถานะพบว่า **ไม่ได้ลงทะเบียนทั้ง LINE และ APP**, **When** batch ประมวลผลเวลา 10:00 น., **Then** ระบบส่ง SMS ที่มีชื่อลูกค้า มูลค่ารางวัล และลิงก์ดาวน์โหลดถูกต้อง และบันทึกประวัติการส่งพร้อม ID เชื่อมโยงกลับรายการรอบแรก
2. **Given** ลูกค้าเข้าเงื่อนไขครบ แต่ผลตรวจพบว่า **ลงทะเบียน LINE แล้วและสถานะบล็อก LINE OA** (isBlockLine = true) และยังไม่มี Ocean Club App, **When** batch ประมวลผล, **Then** ระบบส่ง SMS ให้ลูกค้ารายนี้ (เพราะช่องทาง LINE ใช้ไม่ได้จริงและยังไม่มีแอป)
3. **Given** ลูกค้าเข้าเงื่อนไขเวลา แต่ผลตรวจพบว่า **ลงทะเบียน Ocean Club App แล้ว** หรือ **ลงทะเบียน LINE แล้วโดยไม่บล็อก**, **When** batch ประมวลผล, **Then** ระบบคัดลูกค้ารายนี้ออก ไม่ส่ง SMS
4. **Given** ลูกค้าเข้าเงื่อนไขเวลา แต่กรมธรรม์พ้นสภาพ (ไม่ Inforce) หรือไม่มีเบอร์โทรศัพท์ที่ติดต่อได้ หรือติดเงื่อนไข Do Not Contact ตามประเภทกรมธรรม์, **When** batch ประมวลผล, **Then** ระบบคัดลูกค้ารายนี้ออก ไม่ส่ง SMS
5. **Given** ลูกค้าเคยได้รับ SMS รอบสอง (Ocean Club) ไปแล้ว (มีประวัติ `policy_no` + `policy_type` + เบอร์โทรตรงกัน และมีวันที่ส่งสำเร็จ), **When** batch ประมวลผลรอบถัดไป, **Then** ระบบไม่ส่ง SMS ซ้ำให้ลูกค้ารายนี้
6. **Given** ค่าตั้งต้นจำนวนวันในระบบ config ถูกเปลี่ยนจาก 20 เป็นค่าอื่นโดยแคมเปญยัง Active, **When** batch ประมวลผลรอบถัดไป, **Then** ระบบใช้จำนวนวันใหม่ในการคัดเลือกกลุ่มเป้าหมายโดยไม่ต้องแก้ไขระบบ

### User Story 2 - แจ้งเตือนเมื่อ Batch ล้มเหลว (Priority: P2)

ในฐานะทีม IT Development และกลุ่มผู้ใช้งาน (User) ต้องการได้รับ email แจ้งเตือนทันทีเมื่อ batch ไม่สามารถประมวลผลได้ตามรอบเวลา (เช่น สกัดข้อมูลไม่ได้ สร้างไฟล์ไม่ได้ หรือบริการภายนอกล่ม) เพื่อให้เข้าแก้ไขและซ่อมข้อมูลได้เร็วที่สุด

**Why this priority**: หากรอบส่งล้มเหลวเงียบๆ ลูกค้ากลุ่มนั้นจะไม่ได้รับข้อความและไม่มีใครรู้ — การแจ้งเตือนเป็นกลไกความน่าเชื่อถือหลักของงาน batch

**Independent Test**: จำลองความล้มเหลวระดับรอบ (เช่น ปิดการเข้าถึงแหล่งข้อมูล หรือปิดค่า config แคมเปญทั้งหมด) แล้วตรวจว่า email แจ้งเตือนถูกส่งถึงทีม IT และ User โดยระบุขั้นตอนที่ล้มเหลว

**Acceptance Scenarios**:

1. **Given** batch ไม่สามารถสกัดข้อมูลหรือสร้างไฟล์ CSV ได้ตามรอบเวลา 10:00 น., **When** ระบบตรวจพบความล้มเหลว, **Then** ระบบส่ง email แจ้งเตือนไปยังทีม IT Development และกลุ่มผู้ใช้งานทันที โดยระบุขั้นตอนที่ล้มเหลว
2. **Given** บริการตรวจสอบสถานะการลงทะเบียน (LINE/APP) ไม่ตอบสนองทั้งระบบ, **When** ระบบ retry ครบจำนวนที่กำหนดแล้วยังไม่สำเร็จ, **Then** ระบบหยุดรอบนั้นและส่ง email แจ้งเตือนแบบเดียวกับความล้มเหลวระดับรอบ
3. **Given** วันประมวลผลไม่มีค่า config แคมเปญที่ Active เลย (จำนวนวัน/รางวัล/ลิงก์), **When** batch เริ่มประมวลผล, **Then** ระบบหยุดรอบนั้นทั้งรอบและส่ง email แจ้งเตือน

### User Story 3 - ซ่อมข้อมูลย้อนหลังด้วย Manual Fix (Priority: P3)

ในฐานะผู้ดูแลระบบ (`IT_ADMIN`) ต้องการสั่งรัน batch ย้อนหลังผ่านหน้า Admin Dashboard กลางของ CSMS โดยเลือกช่วงวันที่ที่ต้องการประมวลผลใหม่ได้ เพื่อซ่อมข้อมูลรอบที่ล้มเหลวหรือส่งไม่ครบ

**Why this priority**: จำเป็นต่อการปฏิบัติงานจริงเมื่อเกิดเหตุขัดข้อง แต่ใช้งานเป็นครั้งคราว — flow หลัก (US1) และการรับรู้ปัญหา (US2) ต้องมาก่อน

**Independent Test**: ทำให้รอบวันหนึ่งไม่ถูกประมวลผล แล้วให้ IT Admin สั่ง Manual Fix เลือกช่วงวันที่นั้น — ตรวจว่ารายการค้างถูกประมวลผลและส่ง SMS ครบ โดยไม่ส่งซ้ำให้รายการที่เคยส่งสำเร็จแล้ว

**Acceptance Scenarios**:

1. **Given** รอบ batch ของวันหนึ่งล้มเหลวทั้งรอบ, **When** IT Admin เลือกช่วงวันที่นั้นและสั่ง Manual Fix, **Then** ระบบประมวลผลกลุ่มเป้าหมายของรอบวันนั้นใหม่และส่ง SMS ให้รายการที่ยังไม่เคยส่งสำเร็จ
2. **Given** ช่วงวันที่ที่เลือกมีทั้งรายการที่ส่งสำเร็จแล้วและรายการค้าง, **When** สั่ง Manual Fix, **Then** ระบบส่งเฉพาะรายการค้าง ไม่ส่งซ้ำรายการที่สำเร็จแล้ว

---

### Edge Cases

- ผลตรวจสถานะการลงทะเบียนตอบกลับเป็น Error/Exception (`E13`) หรือค่านอกเหนือเงื่อนไขที่ระบุ (เช่น channel/isBlockLine เป็นค่าว่าง) → skip รายการนั้น + บันทึก log แยกให้ IT Admin ตรวจสอบ/Manual Fix ภายหลัง (แนวทางเดียวกับ CSMS-001)
- ไม่พบ `cardNo` จากหมายเลขกรมธรรม์ → skip + บันทึก sent_status='F' รายรายการ ไม่หยุดรอบ (ตาม FR-007 — แนวทางเดียวกับ 002 EC-005)
- ลูกค้ามีเบอร์โทรทั้ง mobile1 และ mobile2 → ใช้ mobile1 เป็นหลัก หากว่างจึงใช้ mobile2 (ตาม FR-005)
- เบอร์ในประวัติรอบแรกกับเบอร์ปัจจุบันในข้อมูลกรมธรรม์ไม่ตรงกัน (ลูกค้าเปลี่ยนเบอร์ระหว่าง 20 วัน) → ส่งไปเบอร์ปัจจุบัน และบันทึกเบอร์ที่ส่งจริงลงประวัติ (ตาม FR-005, FR-018)
- ลูกค้าลงทะเบียน App ในช่วงเวลาหลังคัดกรองแต่ก่อนส่งจริง → ยอมรับได้ (ตรวจ ณ เวลาประมวลผลเท่านั้น)
- ลูกค้า 1 คนมีหลายกรมธรรม์ที่เข้าเงื่อนไขพร้อมกันในรอบเดียว → จัดกลุ่มตาม (ชื่อ, นามสกุล, เบอร์โทร) ส่งเพียง 1 ข้อความต่อลูกค้าต่อรอบ ตาม FR-013.1 (แนวทางเดียวกับ CSMS-001)
- รายการที่ส่งไม่สำเร็จจาก SMS Gateway → บันทึกสถานะไม่สำเร็จโดยไม่มีวันที่ส่งสำเร็จ เพื่อให้ Manual Fix หยิบกลับมาส่งใหม่ได้ (แนวทาง `sent_status` เดียวกับ CSMS-001)
- ค่า config จำนวนวันถูกเปลี่ยน (เช่น 20 → 15) ทำให้บางกลุ่มถูกข้ามหรือถูกนับซ้ำ → กลไกกันส่งซ้ำต้องป้องกันการส่งซ้ำ ส่วนกลุ่มที่ถูกข้ามซ่อมได้ผ่าน Manual Fix
- รอบแรก (CSMS-001) ของวันอ้างอิงไม่มีข้อมูลเลย (เช่น รอบแรกล้มเหลวเมื่อ 20 วันก่อน) → รอบนี้ไม่มีเป้าหมาย ถือว่าจบรอบปกติ (ไม่ใช่ความล้มเหลว)

## Requirements *(mandatory)*

### Functional Requirements

**การคัดเลือกกลุ่มเป้าหมาย**

- **FR-001**: ระบบ MUST ประมวลผลอัตโนมัติทุกวัน (Daily) เวลา 10:00 น.
- **FR-002**: ระบบ MUST ดึงกลุ่มเป้าหมายจากประวัติการส่ง SMS รอบแรก (CSMS-001 — Welcome New Customer LINE) ที่วันที่ส่งสำเร็จครบกำหนดจำนวนวันที่ตั้งค่าไว้ ณ วันประมวลผล โดยดึงข้อมูลเลขที่กรมธรรม์ ประเภทกรมธรรม์ และเบอร์โทรศัพท์มาตรวจสอบต่อ
- **FR-003**: จำนวนวันที่ครบกำหนด MUST อ่านจากค่า config กลางของแคมเปญ (ประเภท DATE_COUNT ของแคมเปญ CSMS_SNC_NewCustApp ที่สถานะ Active) โดยค่าตั้งต้นคือ 20 วัน และเปลี่ยนค่าได้โดยไม่ต้องแก้ไขระบบ
- **FR-004**: ระบบ MUST ตรวจสอบสถานะกรมธรรม์ให้อยู่ในสถานะมีผลบังคับ (Inforce) เท่านั้น — ORD/PA ใช้สถานะ '1'; IND/GOV ใช้สถานะ '1', '2' หรือ '3' โดยขอบเขตแคมเปญครอบคลุมกรมธรรม์ 4 ประเภท: ORD, IND, GOV, PA (ยืนยันแล้ว — ดู Clarifications)
- **FR-005**: ระบบ MUST คัดเฉพาะลูกค้าที่มีเบอร์โทรศัพท์ติดต่อได้ (mobile1 หรือ mobile2 อย่างน้อยหนึ่งช่องไม่เป็นค่าว่าง) และเบอร์ที่ใช้ส่งจริง MUST เป็นเบอร์ปัจจุบันจากข้อมูลกรมธรรม์ โดยใช้ mobile1 เป็นหลัก หากว่างจึงใช้ mobile2 — เบอร์จากประวัติรอบแรกใช้เพื่อการอ้างอิงเท่านั้น (ยืนยันแล้ว — ดู Clarifications)
- **FR-006**: ระบบ MUST คัดออกลูกค้าที่ติดเงื่อนไข Do Not Contact: กรมธรรม์ ORD ที่มี remark code '1' และกรมธรรม์ IND/GOV/PA ที่มี remark code '4'

**การตรวจสอบสถานะการลงทะเบียน**

- **FR-007**: ระบบ MUST แปลงหมายเลขกรมธรรม์เป็นหมายเลขประจำตัวลูกค้า (cardNo) ผ่านบริการตรวจสอบข้อมูลลูกค้า — หากไม่พบข้อมูล MUST skip รายการนั้น + บันทึกลงประวัติรายกรมธรรม์ด้วย sent_status='F' โดยไม่หยุดรอบ เพื่อให้ IT Admin ตรวจสอบ/Manual Fix ภายหลัง (ยืนยันแล้ว — ดู Clarifications; ปรับจากเอกสารต้นทางที่ระบุเพียง "คัดออก" ตามแนวทาง 002: EC-005)
- **FR-008**: ระบบ MUST ตรวจสอบสถานะการลงทะเบียน 2 ช่องทางต่อลูกค้าแต่ละราย: LINE Ocean Connect (channel LINE) และ Ocean Club App (channel APP)
- **FR-009**: ระบบ MUST ส่ง SMS เฉพาะกรณีต่อไปนี้: (ก) ไม่พบการลงทะเบียนทั้ง LINE และ APP (ผลตรวจ "ไม่พบข้อมูล" — E02 — ทั้งสองช่องทาง) หรือ (ข) พบการลงทะเบียน LINE แต่สถานะบล็อก LINE OA (isBlockLine = true) และไม่พบการลงทะเบียน APP — สถานะ "ไม่พบข้อมูล" ใช้รหัสเดียวกันทั้งสองช่องทาง (ยืนยันแล้ว — ดู Clarifications)
- **FR-010**: ระบบ MUST คัดออกลูกค้าที่พบการลงทะเบียน APP แล้ว หรือพบการลงทะเบียน LINE โดยไม่บล็อก (isBlockLine = false)
- **FR-011**: กรณีผลตรวจเป็น Error/Exception (E13) หรือค่านอกเหนือเงื่อนไขที่ระบุ (เช่น channel/isBlockLine เป็นค่าว่าง) ระบบ MUST skip รายการนั้นทันทีโดยไม่ retry และไม่หยุดทั้งรอบ — บันทึกรายการที่ skip ลงประวัติรายกรมธรรม์ด้วย sent_status='F' (ไม่มีวันที่ส่งสำเร็จ) เพื่อให้ IT Admin ตรวจสอบ/Manual Fix ภายหลัง และประมวลผลรายการถัดไปต่อจนจบ (แนวทางเดียวกับ 002: FR-008/FR-008a)
- **FR-012**: กรณีบริการตรวจสอบสถานะไม่ตอบสนองทั้งระบบ ระบบ MUST retry จำนวนจำกัด (สูงสุด 3 ครั้ง เว้นช่วง) หากยังไม่สำเร็จ MUST หยุดรอบนั้นและแจ้งเตือนตาม FR-019

**การป้องกันการส่งซ้ำ**

- **FR-013**: ระบบ MUST ไม่ส่ง SMS ซ้ำให้รายการที่มีประวัติการส่งรอบสอง (Ocean Club) สำเร็จแล้ว โดยเทียบจากเลขที่กรมธรรม์ + ประเภทกรมธรรม์ + เบอร์โทรศัพท์ ที่มีวันที่ส่งสำเร็จ
- **FR-013.1**: ภายในรอบประมวลผลเดียวกัน หากลูกค้าคนเดียวกัน (ชื่อ-นามสกุล-เบอร์โทรตรงกัน) มีหลายกรมธรรม์เข้าเงื่อนไข ระบบ MUST จัดกลุ่มและส่งเพียง 1 ข้อความต่อลูกค้า โดยเลือกรายการตัวแทนแบบ deterministic ตามแนวทางเดียวกับ CSMS-001 (รายการเก่าสุด, tie-break ด้วยเลขที่กรมธรรม์น้อยสุด) — รายการที่ไม่ถูกเลือกบันทึกประวัติตามแนวทางเดียวกับ CSMS-001 เพื่อให้ reconcile ได้ครบ (ยืนยันแล้ว — ดู Clarifications)

**ข้อความและไฟล์นำส่ง**

- **FR-014**: ระบบ MUST ดึงรูปแบบข้อความจากตารางกำหนดการข้อความ (ประเภทข้อความรหัส '111') และหากไม่พบ MUST ใช้รูปแบบจากตารางประเภทข้อความหลักแทน (fallback)
- **FR-015**: ระบบ MUST แทนค่าตัวแปรในข้อความให้ครบก่อนส่ง: ชื่อผู้เอาประกัน (${var1}), มูลค่าเหรียญรางวัลจาก config ประเภท REWARDS (${var2}) และลิงก์ดาวน์โหลดจาก config ประเภท LINE_LINK (${var3}) — ค่า config ทั้งหมดต้องมาจากแคมเปญที่สถานะ Active
- **FR-015.1**: หากรายการใดไม่พบชื่อผู้เอาประกัน (ค่าที่ใช้แทน ${var1} ว่าง) ระบบ MUST skip รายการนั้น + บันทึกด้วย sent_status='F' — MUST NOT ส่งข้อความที่ไม่มีชื่อลูกค้า (แนวทางเดียวกับ 002: FR-011a)
- **FR-016**: หากวันประมวลผลไม่มีค่า config แคมเปญที่ Active (จำนวนวัน/รางวัล/ลิงก์) ระบบ MUST หยุดรอบนั้นทั้งรอบและแจ้งเตือนตาม FR-019
- **FR-017**: ระบบ MUST สร้างไฟล์นำส่ง (CSV, UTF-8) ตามรูปแบบชื่อไฟล์มาตรฐาน `MKT_CSMS_MKTWelcomeNewCustApp_[YYMMDDhhmmss].csv` (ปีพุทธศักราช 2 หลัก) แล้วนำส่งผ่านช่องทางกลาง (ESB → SMS Gateway)

**การบันทึกประวัติและการเชื่อมโยง**

- **FR-018**: ระบบ MUST บันทึกประวัติการส่ง 2 ระดับ: (ก) ประวัติภาพรวมของรอบ (ข้อความจริงที่ส่ง, ชื่อย่อประเภทข้อความ, เวลาสร้าง) และ (ข) ประวัติรายกรมธรรม์ (เลขที่กรมธรรม์, ประเภท, ชื่อ-นามสกุล, หมายเลขประจำตัว, วันเริ่มสัญญา, วันที่ส่งสำเร็จ, เบอร์ที่ส่งจริง, ข้อความจริง, จำนวน credit ที่ใช้, ผู้สร้าง/แก้ไข) พร้อมสถานะผลการส่งกลับจากช่องทางกลาง (refer_no และสถานะการส่ง) — ประเภทกรมธรรม์ในประวัติ map ตรงตัวจากรหัสคัดกรองครบ 4 ประเภท: 'O'→ORD, 'I'→IND, 'G'→GOV, 'P'→PA (ยืนยันแล้ว — ดู Clarifications และ Assumptions เรื่องอักขระรหัส)
- **FR-018.1**: ประวัติรายกรมธรรม์ของรอบสอง MUST มีการเชื่อมโยง (ID อ้างอิง) กลับไปยังรายการประวัติรอบแรก (CSMS-001) ที่เป็นต้นทางของกลุ่มเป้าหมายรายนั้น
- **FR-018.2**: ระบบ MUST บันทึก**ทุกรายการที่เข้าขั้นตอนส่งหรือถูก skip** ลง `WELCOME_NEW_CUST_APP` — รายการส่งสำเร็จ: `sms_sent_date` = วันที่ส่ง และ `sent_status` = 'S'; รายการไม่สำเร็จ/ถูก skip (FR-011, FR-015.1): `sms_sent_date` = NULL และ `sent_status` = 'F' — ตาราง `WELCOME_NEW_CUST_APP` MUST เพิ่มฟิลด์ `sent_status` เช่นเดียวกับตารางของ 002 เพื่อให้รายงาน reconcile ครบถ้วนและ Manual Fix ค้นรายการค้างจากตารางเดียว โดยกลไกกันส่งซ้ำ (FR-013) ตรวจเฉพาะรายการที่ `sms_sent_date` IS NOT NULL (แนวทางเดียวกับ 002: FR-016a)

**การแจ้งเตือนและการซ่อมข้อมูล**

- **FR-019**: เมื่อเกิดความล้มเหลวระดับรอบ (สกัดข้อมูล/สร้างไฟล์ไม่ได้, ช่องทางนำส่งล่ม, แหล่งข้อมูลผิดพลาด, บริการตรวจสอบสถานะล่ม, ไม่มี config Active) ระบบ MUST ส่ง email แจ้งเตือนไปยังทีม IT Development และกลุ่มผู้ใช้งานทันที ตามรูปแบบมาตรฐานของ batch CSMS เดิม (ผู้รับ/subject/เนื้อหาตาม convention เดิม) โดยระบุชื่อ batch (ตาม convention เดิม เช่น `BATCH-CSMS-WELCOME-APP`) และ**ขั้นตอนที่ล้มเหลว** — เกณฑ์ "ตามเวลาที่กำหนด" ใช้ timeout/SLA มาตรฐานของระบบ monitoring batch CSMS เดิม (แนวทางเดียวกับ 002: FR-018)
- **FR-020**: IT Admin MUST สามารถสั่งรัน Manual Batch ย้อนหลังผ่านหน้า Admin Dashboard กลางของ CSMS โดยเลือกช่วงวันที่รอบ batch ที่ต้องการประมวลผลใหม่ได้ และการรันซ้ำ MUST ไม่ทำให้เกิดการส่งซ้ำแก่รายการที่สำเร็จแล้ว (ตาม FR-013)
- **FR-020.1**: การส่งซ้ำรายการที่ไม่สำเร็จ (sent_status='F') MUST ทำผ่าน Manual Fix **เท่านั้น** — รอบ daily ปกติ MUST NOT ดึงรายการ 'F' ค้างมา retry อัตโนมัติ; บันทึก 'F' คงไว้เพื่อรายงาน/audit และ Manual Fix หยิบเฉพาะรายการที่ sms_sent_date IS NULL ในช่วงวันที่ที่เลือก (แนวทางเดียวกับ 002: FR-020b)
- **FR-020.2**: ช่วงวันที่ซ่อมข้อมูลที่เลือก MUST ตีความเป็น **วันประมวลผลของรอบ batch ที่ต้องการ re-run** — ระบบหากลุ่มเป้าหมายจากประวัติรอบแรก (CSMS-001) ที่วันที่ส่งสำเร็จ = วันที่ที่เลือก ลบด้วยจำนวนวัน N โดยใช้**ค่า N ปัจจุบันจาก config ณ เวลาสั่ง Manual Fix** (ไม่เก็บประวัติค่า config — หาก N เคยถูกเปลี่ยน IT Admin ปรับช่วงวันที่ชดเชยได้ และ FR-013/FR-020 กันการส่งซ้ำอยู่แล้ว) (ยืนยันแล้ว — ดู Clarifications; เทียบเคียง 002: FR-020a): ผลการส่งของแคมเปญนี้ MUST ปรากฏในรายงานสรุปประจำวันกลางของ CSMS ที่ใช้ร่วมกันทุกแคมเปญ — ไม่สร้างรายงานแยกใหม่ (ยืนยันแล้ว — ดู Clarifications) ประกอบด้วย: จำนวนที่ประมวลผลทั้งหมด, ส่งสำเร็จ (sent_status='S'), ไม่สำเร็จ/ถูก skip (sent_status='F'), และจำนวนที่ถูกตัดออกตามเหตุผล (Do Not Contact / ลงทะเบียนแล้ว / ซ้ำในรอบ / เคยส่งแล้ว) พร้อม breakdown ตามประเภทกรมธรรม์ (ORD/IND/GOV/PA) — รายงานนี้เป็นช่องทางเดียวสำหรับติดตามสัดส่วนรายการ 'F' ที่สูงผิดปกติ ไม่มีกลไก threshold alert แยก (แนวทางเดียวกับ 002: FR-021)

### Key Entities

- **รายการกลุ่มเป้าหมายรอบแรก (Welcome LINE Log)**: ประวัติการส่ง SMS รอบแรกจาก CSMS-001 — ต้นทางของกลุ่มเป้าหมาย ระบุเลขที่กรมธรรม์ ประเภท เบอร์โทร และวันที่ส่งสำเร็จที่ใช้นับวันครบกำหนด
- **กรมธรรม์ (Policy)**: ข้อมูลหลักจากคลังข้อมูลกลาง — สถานะผลบังคับ, ประเภท (ORD/IND/GOV/PA), เบอร์โทรศัพท์, ชื่อผู้เอาประกัน, วันเริ่มสัญญา
- **รายการระงับการติดต่อ (Do Not Contact)**: เงื่อนไข remark ระดับกรมธรรม์ที่ห้ามติดต่อลูกค้า แยกตามประเภทกรมธรรม์
- **สถานะการลงทะเบียนลูกค้า (Registration Status)**: ผลตรวจจากบริการภายนอกต่อ cardNo ใน 2 ช่องทาง (LINE / APP) พร้อมสถานะการบล็อก LINE OA
- **ค่า config แคมเปญ (Campaign Config)**: ค่ากลางของแคมเปญ CSMS_SNC_NewCustApp — จำนวนวันครบกำหนด (DATE_COUNT), มูลค่ารางวัล (REWARDS), ลิงก์ดาวน์โหลด (LINE_LINK) พร้อมสถานะ Active
- **แม่แบบข้อความ (SMS Template)**: รูปแบบข้อความประเภทรหัส '111' พร้อมตัวแปร ${var1}–${var3} และกลไก fallback
- **ไฟล์นำส่ง (Interface File)**: ไฟล์ CSV UTF-8 ตามรูปแบบชื่อมาตรฐาน ที่ส่งเข้าช่องทางกลางเพื่อนำส่ง SMS
- **ประวัติการส่งรอบสอง (Welcome App Log)**: ประวัติรายกรมธรรม์ของแคมเปญนี้ — ใช้กันส่งซ้ำ, ตรวจนับผล, และเชื่อมโยงกลับรายการรอบแรกด้วย ID อ้างอิง — รวมฟิลด์ `sent_status` ('S'=สำเร็จ, 'F'=ไม่สำเร็จ/ถูก skip) เพิ่มจากโครงสร้างในเอกสารต้นทาง ตามแนวทาง 002 (FR-018.2) — **ต้องสร้างใหม่ (เฉพาะ batch นี้)**
- **ประวัติภาพรวม (Campaign Log)**: บันทึกระดับรอบของข้อความที่จัดส่งจริง

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ลูกค้าที่เข้าเงื่อนไขครบทุกข้อ ณ รอบประมวลผล ได้รับ SMS ภายในรอบวันนั้น 100% (ยกเว้นรายการที่ถูก skip โดยมีบันทึกเหตุผลครบ)
- **SC-002**: ลูกค้า 1 ราย (ชื่อ-นามสกุล-เบอร์โทรเดียวกัน) ได้รับ SMS แคมเปญนี้ไม่เกิน 1 ข้อความตลอดอายุแคมเปญ — จำนวนการส่งซ้ำ (ทั้งภายในรอบ ข้ามรอบ และจาก Manual Fix) = 0 รายการ
- **SC-003**: ไม่มี SMS ถูกส่งถึงลูกค้าที่ลงทะเบียน Ocean Club App แล้ว, ลงทะเบียน LINE โดยไม่บล็อก, ติด Do Not Contact หรือกรมธรรม์พ้นสภาพ — อัตราส่งผิดกลุ่มเป็น 0
- **SC-004**: ทุกข้อความที่ส่งมีชื่อลูกค้า มูลค่ารางวัล และลิงก์ครบถ้วนถูกต้อง 100% (ไม่มีตัวแปรค้างรูปแบบ ${var} หลุดถึงลูกค้า)
- **SC-005**: ความล้มเหลวระดับรอบทุกกรณีมี email แจ้งเตือนถึงทีม IT และ User ภายใน 15 นาทีนับจากตรวจพบ
- **SC-006**: ทุกรายการที่ส่งสำเร็จตรวจสอบย้อนกลับได้ถึงรายการรอบแรก (CSMS-001) ที่เป็นต้นทาง 100%
- **SC-007**: รอบที่ล้มเหลวสามารถซ่อมด้วย Manual Fix ให้ลูกค้ากลุ่มที่ค้างได้รับ SMS ครบถ้วนโดยไม่เกิดการส่งซ้ำ

## Assumptions

- **สืบทอดข้อยุติจาก CSMS-001**: แคมเปญนี้เป็น batch พี่น้องใน CSMS module เดียวกัน จึงใช้ข้อยุติเดิมของ `specs/002-batch-welcome-new-customer-line-sms` โดยอนุโลม ได้แก่ การบันทึกรายการไม่สำเร็จด้วย `sent_status='F'` + วันที่ส่งสำเร็จเป็นค่าว่าง, retry บริการภายนอกสูงสุด 3 ครั้ง, email แจ้งเตือนครอบคลุมความล้มเหลวระดับรอบทุกขั้นตอนในรูปแบบเดียวโดยระบุขั้นตอน, เกณฑ์ timeout/SLA ใช้ค่ามาตรฐานของระบบ monitoring batch CSMS เดิม, Manual Fix ใช้หน้า Dashboard กลางที่มีอยู่, และรายการ 'F' ไม่ retry อัตโนมัติในรอบ daily (ซ่อมผ่าน Manual Fix เท่านั้น)
- **การนับวันครบกำหนด**: ใช้แบบตรงวัน (วันที่ส่งรอบแรก + N วัน = วันประมวลผล) เพื่อให้แต่ละรอบมี cohort ชัดเจน — กลุ่มที่พลาดรอบ (เช่น batch ล้มเหลว) ซ่อมผ่าน Manual Fix ตามแนวทางเดียวกับ CSMS-001
- **อักขระรหัสประเภทกรมธรรม์ (ยืนยันแล้วจาก 002)**: spec 002 (Q-001 Closed, A-005) ยืนยันรหัสเป็น**ตัวอักษร**: ORD='O', IND='I', GOV='G', PA='P' — สอดคล้องกับคำตอบ clarification ของ feature นี้ ('O'→ORD, 'I'→IND, 'G'→GOV, 'P'→PA); หมายเหตุ: ในการตรวจ Do Not Contact 002 จัดกลุ่ม 'G' ร่วมกับ IND ซึ่งไม่ขัดกัน เพราะ IND/GOV ใช้ remark_code '4' เหมือนกัน (FR-006) — เลข '0'/'1' ที่ปรากฏในเอกสารต้นทางของ feature นี้ถือเป็นความคลาดเคลื่อนจากการแปลง PDF
- **ฟิลด์ลำดับ #7 ที่หายไป**: ตารางประวัติรายกรมธรรม์ในเอกสารต้นทางข้ามลำดับฟิลด์ #7 — สันนิษฐานว่าเป็นความตกหล่นจากการแปลง PDF จะตรวจเทียบกับโครงสร้างตารางจริงของ CSMS-001 (`WELCOME_NEW_CUST_LINE`) ในขั้น plan
- **credit_amount**: ใช้ logic การคำนวณเดิมของ CSMS (นอก scope feature นี้)
- **ปริมาณข้อมูล**: กลุ่มเป้าหมายประมาณ 1,000 รายต่อวัน (ยืนยันแล้ว — ดู Clarifications) ใช้เป็นฐานออกแบบการเรียกบริการตรวจสอบสถานะแบบชุด ขนาดไฟล์นำส่ง และการทดสอบ scale ในขั้น plan
- **ระบบต้นน้ำพร้อมใช้**: batch CSMS-001 ทำงานปกติและมีข้อมูลประวัติครบถ้วนก่อนแคมเปญนี้เริ่ม, คลังข้อมูลกรมธรรม์และบริการตรวจสอบสถานะลูกค้าพร้อมให้บริการในช่วงเวลาประมวลผล
- **ไม่มีการเปลี่ยนแปลงข้อความระหว่างรอบ**: ข้อความและมูลค่ารางวัลที่ส่งยึดตามค่า config ณ เวลาประมวลผลของรอบนั้น
