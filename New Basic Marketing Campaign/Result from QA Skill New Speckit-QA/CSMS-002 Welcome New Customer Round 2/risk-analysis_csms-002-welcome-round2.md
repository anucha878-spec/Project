### เหตุผลของคะแนนที่สูงผิดจากที่คาด (justification)

- **FR-017 = 18/18 (P0 สูงสุด)** — ไม่ใช่เพราะ logic ซับซ้อน แต่เพราะ **ยังไม่มีใครรู้ว่าข้อกำหนดคืออะไร**
  เอกสาร 2 ฉบับขัดแย้งกันโดยตรง และ `plan.md` **อ้างการแก้ spec ที่ไม่เคยเกิดขึ้น** (CF-001) → Hist-defect=3
  ไม่ว่าจะตัดสินทางใด อีกทางหนึ่งคือ **rework ทั้งก้อน**: ถ้า CSV ชนะ → ต้องสร้าง infra ใหม่ที่ `plan.md` ยืนยันว่าไม่มี (Complexity=3)
  ถ้า API ชนะ → ต้องแก้ spec 5 จุด Data-sens=3 เพราะทั้งสองทางขนถ่าย PII (ไฟล์ ~1,000 ราย / call ละ 1 ราย)
- **FR-002 = 15 (P0)** — Integration=3 เพราะเป็น **จุดพึ่งพา CSMS-001 เพียงจุดเดียว** (CF-008); Complexity=3 เพราะ `plan.md`
  ระบุว่าเป็น **2-step SQL ใหม่ทั้งหมด** ที่อ่าน `ili_policy_master`/`ili_policy_remark` โดยตรง — ต่างจาก 002 ที่อ่าน `ili_application_osp_*`
- **FR-018 / FR-015 / FR-006 / FR-005 / FR-007 = P0/P1 จาก Data-sens=3** — ทุกข้อแตะ **cardNo (เลขประจำตัวประชาชน)**
  หรือเบอร์มือถือ + ผูกกับ constitution MUST-principle (I/II/IV) ที่ `plan.md` **เลือกยกเว้น** แทนที่จะปิดช่องว่าง
  ตามกฎ §4.5 "Data-sensitivity=3 ร่วมกับ MUST-principle ⇒ P0 อัตโนมัติ" → **FR-018 และ FR-006 ยกเป็น P0 แม้คะแนนรวม 13 และ 11**
  (FR-015, FR-005, FR-007 คงไว้ที่ P1 เพราะ MUST-principle ที่เกี่ยวข้องถูกคลุมด้วย TC ที่ไม่มีข้อขัดแย้งค้าง)
  **FR-013 ยกเป็น P0** เช่นกัน — เพราะกลไกกันส่งซ้ำใช้ PII (ชื่อ+เบอร์) เป็นคีย์ และ SC-002 กำหนดว่าการส่งซ้ำต้อง = 0
- **FR-014 / FR-020.2 = Hist-defect 3** — ทั้งคู่มี **ข้อบกพร่องเชิงเอกสารที่ยังไม่ปิด**: FR-014 มีรหัสขัดแย้ง '111' vs `114` (CF-004);
  FR-020.2 **รวม 2 requirement ในข้อเดียว** และครึ่งรายงานไม่มีเลข FR (CF-003)
- **FR-001 = Hist-defect 3 แต่ Band P2** — cron ของ 002 ผิดจริงและไม่เคยถูกแก้ (CF-009) แต่เป็นข้อบกพร่องที่
  **ตรวจพบง่ายมากด้วย unit test เดียว** (TC-002) ผลกระทบทางธุรกิจจำกัด (ส่งผิดเวลา ไม่ใช่ส่งผิดคน) จึงไม่ยกขึ้น P1
