## กฎการให้คะแนนที่ใช้

- 6 แกน แกนละ 0–3 · Band: **P0** >= 14 **หรือ Data-sensitivity = 3 คู่กับ MUST-principle** · **P1** 10–13 · **P2** 5–9 · **P3** 0–4

### เหตุผลของคะแนนที่ควรอธิบาย

- **FR-012 (12 คะแนน) และ FR-013 (10 คะแนน) ถูกยกเป็น P0 ทั้งที่คะแนนรวมต่ำกว่า 14** — เพราะกฎ "Data-sensitivity = 3 คู่กับ MUST-principle" **บังคับ** ไม่ใช่ดุลพินิจ FR-012 เก็บ `cardNo`/ชื่อ/เบอร์ลงตารางใหม่ถาวร และ FR-013 ส่ง email **ออกนอกระบบ** ที่อาจพา stack trace ซึ่งมี PII ออกไปด้วย — ทั้งสองเป็นทางรั่วของ PII ที่แท้จริง
- **FR-008b / FR-008c ได้ Hist-defect = 3** — ตรรกะ dedup ที่ผูกกับ "ปี" มีจุด off-by-one ที่ข้ามปีปฏิทิน (TC-049) และทับซ้อนกับกฎ 29 ก.พ. (TC-050) และ **ไม่มี plan.md บอกว่ามีกลไกกันรันซ้อนหรือไม่** (F-009)
- **FR-003a ได้ Complexity 3 + Hist-defect 3 แต่ Integration 0** — ตรรกะวันที่ล้วน ซับซ้อนและพลาดง่ายมาก แต่ไม่แตะระบบภายนอก จึงลงเอยที่ P1 ไม่ใช่ P0 **ทั้งนี้ยังต้องทดสอบหนักที่สุดในมิติ Edge** (6 TC) — ระดับ P1 ไม่ได้ลดปริมาณการทดสอบ เพียงจัดลำดับว่ารันทีหลัง P0
- **FR-010 / FR-010b ได้ Change-freq = 2** — สูงผิดปกติสำหรับ requirement ที่เพิ่งเขียน เพราะ **มีความน่าจะเป็นสูงมากที่จะถูกลบทิ้งทั้งข้อ** จากข้อขัดแย้ง CSV vs API (F-002)
